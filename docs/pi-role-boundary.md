# Pi-first 单人叙事 MUD：Pi 与 Engine 的职责边界

## 1. 产品定位

`mud-pi` 不是一个试图用代码穷举全部规则、自动运行世界生态的传统 MUD Server，也不是只有聊天记录、没有客观状态约束的 AI 角色扮演聊天。

它是一个 **以持久 Pi Session 为主持人，Engine 为权威状态内核的单人叙事 RPG 框架**：

```text
玩家自由输入
  → 持久 Pi DM 理解意图、裁定开放情境、推进叙事
  → Pi 提交受控 Proposal
  → Engine 校验权限、引用、范围、时效和状态不变量
  → 提交 WorldEvent 并演化 WorldState
  → Pi 根据已提交事实叙述结果
```

核心目标不是让 Engine 自动扮演 DM，而是给 Pi 提供：

1. 可持续的客观世界；
2. 不会被自然语言随意破坏的状态边界；
3. 可查询、可保存、可重放的关键事实；
4. 足够少而通用的行动结果协议；
5. 让 Pi 能长期主持同一场游戏的持久 Session。

本项目明确保持 **单人游戏**。不建设多玩家并发、PvP、公会、房间广播、在线 Builder、区域分片或多人权限系统。

### 桌游心智模型

最准确的类比不是“AI 驱动的自动 MUD”，而是一场由 Pi 主持的单人桌游：

| mud-pi 组件 | 桌游中的对应物 |
|---|---|
| 持久 Pi DM Session | 真人 DM 及其长期主持笔记 |
| 重要 NPC Pi Session | DM 为关键角色保存的独立人物笔记与扮演视角 |
| 世界包 | 规则书、冒险模组、职业/角色卡、道具卡和遭遇表 |
| `WorldState` | 角色纸、地图、棋子位置、计数器、状态标记和战役记录表 |
| Engine | 数字桌面、卡牌管理员、骰塔、记录员和规则边界检查器 |
| WorldEvent Journal | 按顺序记录已经落桌并确认的战役日志 |
| CLI/TUI/Telnet | 玩家查看桌面和向 DM 描述行动的界面 |

在桌游中，角色卡和道具卡不会自己推进故事，记录本也不会自动安排 NPC 作息；真人 DM 根据玩家决定和当前桌面状态主持游戏。`mud-pi` 同样应由 Pi 主持推进，Engine 只负责让桌面组件、计数器和已确认事实保持一致。

因此 Engine 的目标不是“知道如何处理所有游戏行为”，而是提供一组可靠的 GM 操作：移动棋子、增减计数器、发放或消耗卡牌、放置状态标记、揭示地图、记录事实和标记目标。Pi 决定何时、为何以及以什么叙事含义执行这些操作。

---

## 2. Pi 的定位

Pi 是游戏中的 **持久主持人、语义裁判和内容导演**，而不是单纯的文字润色器。

Pi 应负责：

- 理解玩家没有预定义命令的自由行动；
- 判断行动是否符合当前情境；
- 裁定陷阱、机关、调查、潜行、说服、欺骗、追逐等开放交互；
- 决定成功、部分成功、失败以及代价；
- 生成符合世界观的场景、线索、临时 NPC 和可交互物品；
- 推进剧情、任务语义、关系变化和 StoryOutcome；
- 控制非重要 NPC；
- 为重要 NPC 提供可感知事实，并由其独立 Pi Session 作出角色化决定；
- 决定何时不发生特殊事情，避免每个房间都出现机关、奖励或战斗；
- 将已经提交的权威结果转化为自然、沉浸的叙述。

Pi 不只是从规则表中选按钮。对于“撬锁”“用铁杆卡住齿轮”“假装是维修员”“把酒洒在地上测试坡度”等长尾行为，不应为每一种行为编写专用 Engine 系统。

---

## 3. Engine 的定位

Engine 是 **权威状态守门人、数字桌面和提交内核**，不是自动 DM，也不是取代 DM 的完整规则裁判。

它更接近桌游中的角色纸、卡牌、棋子、计数器、骰塔和记录员：能够拒绝“把不存在的卡放进背包”“同一棋子同时在两个房间”“把计数器改成非法值”，但不会自行决定玩家的谎言是否精彩，或一块地砖是否应该触发陷阱。

Engine 应负责：

- 保存玩家、重要 NPC、房间、物品、参数、目标和结果等客观状态；
- 保证实体引用合法；
- 保证物品只有一个权威位置；
- 保证不可携带物不能进入背包；
- 保证装备、使用和消耗符合世界包数据；
- 保证参数变化有限、在允许范围内并触发 thresholds；
- 保证客观死亡、失能和生命周期一致；
- 保证 Agent 不能直接改写独立 NPC 或其他受保护实体；
- 保证 stale Proposal、重复奖励和越权状态变化被拒绝；
- 提供确定性 seed、骰子和可信世界脚本；
- 原子提交状态变化；
- 从 committed WorldEvent 投影 UI、NPC 感知和存档记录；
- 在崩溃后恢复已提交的权威事实。

Engine 不应该判断：

- 某句谎言是否有说服力；
- 一块松动砖头后面是否应该有暗格；
- 玩家提出的创造性方案是否聪明；
- NPC 是否被情感打动；
- 陷阱在当前叙事中是否触发；
- 搜索失败是否仍获得模糊线索；
- 剧情下一幕应该出现什么。

这些属于 Pi 的语义裁定。

## 4. GM 回合协议：先裁定落桌，再向玩家确认

桌游中，DM 不会先宣布“你获得了圣剑”，再发现桌上没有这张卡。数字流程也应避免把尚未提交的提案直接展示成既成事实。

目标回合应分为：

```text
1. 玩家描述行动
2. Pi DM 进行语义裁定，输出候选叙事结果 + GM 桌面操作 Proposal
3. Engine 结算 Proposal
4. 若全部接受：提交 WorldEvent，再向玩家展示与 committed facts 一致的叙述
5. 若有拒绝/修正：暂不展示矛盾叙述，将结构化反馈发回同一 DM Session
6. Pi 在有限重试预算内修正桌面操作和叙述
7. 仍无法合法提交时，使用本地沉浸式 fallback，并明确只采用已提交事实
```

实现上可以缓存 Pi 的 `<NARRATION>`，直到 Settlement 成功；不能先输出叙述再尝试修改 State。为控制成本，默认最多进行一次同回合修正请求，且纯叙事、无状态 Proposal 的回合不需要第二次调用。

这使 Pi 仍然是 DM，但 Engine 像桌面管理员一样只承认已经真正放上桌面的卡牌、标记和计数器变化。

## 5. 边界原则：代码实现名词和不变量，Pi 解释动词和意义

推荐判断标准：

> 如果缺少代码约束会导致存档自相矛盾、复制物品、越权修改、无法恢复或 UI 无法展示，就由 Engine 实现。
>
> 如果规则主要用于理解开放语境、判断合理性或选择有趣后果，就交给 Pi。

### Engine 维护的“名词”

- Entity / NPC / Player
- Room / Exit
- Item / Inventory / Equipment
- Parameter
- Trait
- Applied condition（通用、无语义容器）
- Objective / StoryOutcome
- WorldFact / Flag
- Relationship value（若剧本需要）
- WorldEvent / Revision

### Pi 解释的“动词”

- 搜索、试探、威胁、安慰、欺骗、说服；
- 绕过、拆除、破坏、攀爬、伪装；
- 对陷阱或机关采用任意创造性方案；
- 使用环境、组合物品、临时制作；
- 判断关系和剧情含义；
- 选择成功程度、代价和后续线索。

Engine 只需要接收这些动词最终造成的受控结果，而不需要理解所有动词本身。

## 6. 典型场景的职责划分

### 6.1 陷阱

不建设完整的 `TrapDefinition → Trigger → Detection → Disarm → Reset` 自动模拟器。

Pi 可以根据场景、玩家描述、角色状态和世界规则直接裁定：

- 没有触发；
- 发现陷阱；
- 部分发现，只察觉异常；
- 成功绕过；
- 触发并受到代价；
- 触发但保护了某件物品；
- 制造噪音并惊动 NPC。

但 Pi 不能直接写 State。它只能提出类似：

```json
{
  "factsAdded": ["玩家已经发现石门前的压力板"],
  "parameterChanges": [{ "entityId": "player", "parameterId": "vigor", "delta": -2 }],
  "conditionsApplied": [{ "entityId": "player", "conditionId": "bleeding", "duration": 3 }],
  "itemsCreated": [],
  "narrativeSignals": ["loud_noise"]
}
```

Engine 校验参数、condition ID、范围、实体、时效和权限后提交事实。

### 6.2 门、锁和机关

除非某个剧本明确需要高频、可重复、可查询的门状态，否则不为所有门建立专用对象系统。

普通一次性场景可以由：

- `WorldFact`；
- 房间描述更新；
- 出口开放/关闭 Proposal；
- 通用 flag；

表达。

只有当一个状态必须被 Engine 持续约束（例如关闭的出口绝不能通过）时，才将它提升为权威结构字段。

### 6.3 检定

不强制所有自由行动都进入统一技能表。

世界包可以选择：

- 由 Pi 直接语义裁定；
- 由 Pi 请求一次 Engine 确定性骰子；
- 由可信世界脚本计算；
- 对关键冲突使用已有结构化 Conflict Resolver。

就像真人 DM 可以决定一次行动无需掷骰、指定一次检定，或查阅模组中的固定流程一样，骰子和世界脚本都是 Pi 可调用的桌面工具，不是每个动作必须经过的自动流水线。Engine 负责诚实地产生和记录骰子结果，Pi 负责决定为何掷骰以及结果在故事中意味着什么。

### 6.4 任务

Engine 负责可以客观验证的 Objective：

- 到达地点；
- 与指定 NPC 交谈；
- 获得物品；
- 击败实体；
- 后续少量通用组合条件。

Pi 负责：

- 判断开放式任务是否在语义上完成；
- 决定任务是否出现分支、失败或转化；
- 判断 NPC 是否认可玩家的完成方式；
- 判断是否发放奖励；
- 生成符合剧情的任务描述和后续。

不要把每个剧情步骤编码成自动 Quest FSM。只有必须被 UI 展示、存档恢复或规则引用的里程碑才进入 Objective。

### 6.5 NPC

重要 NPC 使用独立持久 Pi Session，保存主观记忆和人格连续性。

Engine 只保存其客观事实：

- 所在房间；
- 生命周期；
- 参数；
- 持有物；
- 公开关系值或已确认承诺；
- 是否为任务相关实体。

不建设复杂自动日程、效用 AI、行为树或世界生态模拟。NPC 在与玩家或重要事件相关时被唤醒；无关时不需要后台持续行动。

### 6.6 战斗与冲突

关键冲突继续由世界脚本进行确定性、可验证的一次性结算，避免 Pi 任意制造数值结果。

但 Pi 决定：

- 当前是否真的进入冲突；
- 玩家的自由描述对应哪种冲突意图；
- 是否允许交涉、逃跑或创造性绕过；
- 已提交结果在剧情中意味着什么。

并非所有暴力描述都必须进入完整战斗脚本；低风险、纯叙事性的动作可由 Pi 裁定后提交有限结果。

## 7. 当前裁定出口与剩余最小能力

框架已经具备以下通用链路：

```text
ActionIntent
  → Entity Reference Resolution
  → Pi / Deterministic Adjudication
  → CanonicalTableOperation
  → Proposal → Decision → WorldEvent
  → Atomic Commit → evolve()
  → Public Projection / Outbox
  → NarrativeClaim Verification
```

已经落地的桌面操作包括：

- 记录权威事实；
- 创建房间、NPC 和物品；
- 开放或修改出口；
- 移动玩家和受控 NPC；
- 转移、装备和消耗物品卡；
- 调整世界定义参数；
- 应用、移除和过期 Condition；
- 完成允许 Pi 裁定的 Objective；
- 提议世界包声明的 StoryOutcome；
- 发出可感知信号。

Pi 的候选操作被拒绝、调整或与叙述声明不一致时，Runtime 会进行一次有界叙述修正；修正失败则使用 committed-state fallback。

下一步只补充真实游玩反复需要的最小名词：

### P0：SceneObject

用于门、机关、祭坛和其他需要跨回合查询或改变的环境对象。只保存 ID、别名、位置、能力、状态和生命周期，不建设完整机关模拟器。

### P0：稳定 WorldFact key

为需要被 Objective、Outcome 或脚本引用的事实增加稳定 key。自然语言 text 仍用于展示，不建设完整知识图谱。

### P1：更稳定的实体和地点引用

补充 Room/NPC aliases、目的地导航和有界澄清，但不把自然语言重新缩减成固定命令菜单。

### P1：必要时才增加新的世界变量

只有真实世界包证明 Fact、Objective、Condition、Item 和 Outcome 无法可靠表达某项跨回合状态时，才增加轻量变量或时钟。不会预先建设天气、刷新、NPC 排班和自动生态系统。

## 8. 明确不做或暂缓的系统

为了保持 Pi-first 和单人定位，以下内容不进入近期通用 Engine：

- 多玩家和网络并发；
- PvP、公会、组队匹配；
- 自动 NPC 日程和离线世界模拟；
- 怪物刷新和传统刷怪循环；
- 完整门锁陷阱模拟器；
- 为每种技能、法术和环境动作写专用命令；
- 通用制作、采集和经济模拟；
- 自动生成无限任务；
- 自动运行的世界生态；
- 将所有叙事事实结构化；
- 强制所有行为掷骰；
- 用行为树替代 Pi NPC；
- 把传统 D&D 规则硬编码进 Engine。

世界包未来可以按需增加可信脚本，但不会迫使所有世界采用这些系统。

---

## 9. 何时应该把 AI 行为升级为代码机制

只有满足以下一个或多个条件时才升级：

1. **状态一致性**：AI 裁定可能造成复制、越权或矛盾；
2. **高频重复**：同一规则在每局大量发生，Prompt 裁定成本高且不稳定；
3. **跨回合持续**：结果必须被可靠保存、过期或恢复；
4. **UI 查询**：玩家需要随时查看精确状态；
5. **世界包规则明确**：作者要求确定性、公平或可复现；
6. **后续规则依赖**：其他系统必须引用该事实；
7. **存档与重放**：崩溃恢复必须准确重建。

否则默认交给 Pi。

例子：

| 内容 | 默认归属 | 升级为代码的条件 |
|---|---|---|
| 墙后是否有暗格 | Pi | 暗格中物品被创建后必须注册 |
| 是否发现陷阱 | Pi | 世界包要求确定性检定 |
| 受到 3 点伤害 | Engine commit | 始终是权威参数变化 |
| 玩家是否说服 NPC | Pi | 结果形成长期关系 flag 时提交 Engine |
| 门看起来很沉 | Pi | 无需结构化 |
| 门是否阻止移动 | Engine | 因为影响合法移动 |
| NPC 是否喜欢玩家 | Pi Session | 若商店价格或任务依赖它，则保存关系值 |
| 一杯酒让玩家放松 | Pi | 若效果跨回合，则应用 condition |
| 剧情出现一把剑 | Pi 创作 | 可拾取后必须创建权威 Item |

---

## 10. 最终边界

一句话定义：

> Pi 决定“这个世界在当前情境下意味着什么、应该发生什么”；Engine 决定“这些结果能否成为权威事实，以及如何安全地成为事实”。

理想状态不是 Engine 自动运行整个 MUD，而是：

```text
Engine 足够小：只维护关键事实和不变量
Pi 足够强：可以主持绝大多数开放交互
世界包足够明确：提供主题、参数、模板、边界和可信脚本
协议足够稳定：Pi 的自由裁定能安全落地
```

当这四点完成后，后续内容推进主要应由世界包作者和持久 Pi DM 发挥，而不是继续无限扩张通用 Engine。
